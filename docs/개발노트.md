# 개발 노트 — 장치 HL7 v2를 FHIR로 바꾸는 게이트웨이 2주 만에 만들기

> 의료 장치 데이터를 표준으로 번역해 임상시스템에 보내는 흐름을, 직접 손으로 짜보며 배운 것들. 블로그/스레드용 초안.

## 왜 이걸 만들었나

내 강점은 **장치 통신**(TCP/UART/CAN)이다. 비어 있던 건 **의료 상호운용 표준**(HL7/FHIR) 한 겹. 그래서 *장치에서 나온 생체신호를 받아 표준으로 변환해 임상시스템에 보내는* 흐름을 통째로 만들어보기로 했다. 이건 Philips Capsule·GE·Siemens 같은 회사의 **통합 엔지니어가 실제로 하는 일** 그 자체다.

목표는 거창하지 않았다. **"못생겨도 끝까지 한 번 흐르게"**(v0) → 그 다음 단단하게.

## 무엇을 만들었나

```
[Device Simulator] ──HL7 v2 ORU^R01 / MLLP──▶ [Gateway] ──FHIR REST──▶ [HAPI 서버]
   생체신호 4종 송출                       파싱 → 변환 → POST       (공개 테스트 서버)
```

- **Simulator** — 병상 모니터를 흉내내 심박·SpO2·체온·혈압을 HL7 v2.5.1 **ORU^R01** 메시지로 만들어 **MLLP**로 송출
- **Gateway** — MLLP 수신 → NHapi로 ORU 파싱 → FHIR Observation으로 변환 → HAPI에 POST하고 부여 id 회수
- 표준 코드: LOINC(심박 8867-4 등) + UCUM 단위. 스택은 .NET 8 / NHapi / Firely SDK.

## 배운 것 (핵심 4개)

### 1. "장치는 분리해 보내고, FHIR는 묶기를 요구한다"
가장 인상 깊었던 지점. **혈압**은 HL7 v2에서 수축기(8480-6)·이완기(8462-4)·평균(8478-0)이 **각각 OBX 세그먼트로 분리**돼 흐른다. 그런데 FHIR vital-signs 규약은 이걸 **하나의 Blood pressure panel(85354-9)에 component로 묶도록** 요구한다.

즉 게이트웨이가 할 일은 "OBR(오더) 묶음 단위로 흩어진 OBX를 모아 panel로 합치는 것" — 이게 바로 **인터페이스 엔진(Mirth/OIE)이 실제로 하는 변환**이다. 처음엔 "혈압을 어떻게 한 줄에 담지?"로 고민했는데, 실제 HL7을 들여다보니 *애초에 한 줄이 아니었다*. 표준이 현실을 가르쳐준 셈.

### 2. TCP는 메시지가 아니라 바이트 스트림이다
소켓에서 한 번 `read`한 게 메시지 하나와 일치하지 않는다. 그래서 HL7 v2는 **MLLP**라는 프레이밍을 쓴다 — 시작 바이트 `<VT>`(0x0B), 끝 `<FS><CR>`(0x1C 0x0D)로 메시지 경계를 표시. 한 번의 수신에 여러 메시지나 잘린 메시지가 섞여 올 수 있어서, **내부 버퍼에 누적하며 경계를 직접 찾아** 잘라내야 했다(`MllpReader`). 장치 통신 하던 사람에겐 익숙한 "프레이밍" 문제가 의료 표준에도 그대로 있었다.

### 3. 의존성 경계를 처음부터 그어두면 흔들리지 않는다
`Core`(도메인 모델 + LOINC 코드표)는 **HL7도 FHIR SDK도 참조하지 않게** 했다. HL7 파싱은 Gateway, HL7 생성은 Simulator, FHIR 매핑은 Gateway에만 격리. 덕분에 처음에 임의 파이프 포맷으로 짰다가 **진짜 HL7 v2로 통째로 갈아끼울 때** Core는 거의 안 건드렸다. "입력 포맷과 출력 포맷 사이에 중립 도메인을 둔다"는 게 통합 시스템에선 특히 값졌다.

### 4. 라이브러리 메이저 버전의 함정
- **Firely 6.x**: 구 `FhirJsonSerializer`/`SerializerSettings`가 폐기되고 System.Text.Json 기반 `JsonSerializerOptions().ForFhir(...)`로 옮겨졌다. 구버전 예제를 그대로 쓰면 컴파일이 안 된다.
- **네임스페이스 충돌**: `Hl7.Fhir.Model.Task`(FHIR의 Task 리소스)와 `System.Threading.Tasks.Task`가 부딪혀서 `using` 한 줄 때문에 빌드가 깨졌다. 정규명으로 우회.

이런 건 검색해도 잘 안 나오고, **초보면 한참 막혔을** 지점이다.

## "AI 코딩 도구로 초보가 가능한가?" — 내 몸으로 검증

- **AI가 잘 떠먹여준 것**: 소켓 서버/클라이언트 보일러플레이트, NHapi·Firely API 사용법 탐색, 에러 핸들링 패턴, MLLP 프레이밍 바이트 처리, README 초안. 이런 "타이핑량 많고 검색하면 나오는" 일은 압도적으로 빨라졌다.
- **사람이 쥐어야 했던 것**: "혈압을 어떻게 표현할까"(프로토콜 설계), "와이어를 진짜 HL7로 바꾼다"(아키텍처 결정), 의존성 경계를 어디 그을지. **"왜 이렇게"는 위임이 안 됐다** — 오히려 AI가 여러 선택지를 펼쳐주면 그중 *고르는* 판단이 핵심이었다.
- **초보가 막혔을 지점**: 위 라이브러리 버전 함정, 네임스페이스 충돌, "TCP는 스트림이라 프레이밍이 필요하다"는 개념 자체. AI가 풀어주긴 했지만, *질문을 제대로 던지려면* 그 개념을 먼저 알아야 했다.

결론: AI는 **속도를 몇 배 올려주지만, 설계 판단과 도메인 이해를 대체하진 않는다.** "초보 3개월 앱 배포"의 병목은 타이핑이 아니라 *무엇을 만들지 아는 것*이었다.
