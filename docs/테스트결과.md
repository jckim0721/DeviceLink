
통합 테스트는 `SkippableFact`로 **서버 미접속 시 자동 건너뜀**(`metadata` 헬스체크 실패 → skip). 단위 테스트는 네트워크에 의존하지 않아 항상 결정론적.

## 통합 테스트 (3) — 실제 HAPI POST·되조회

- `Full_pipeline_posts_scalar_and_reads_back` — ORU 생성→파싱→매핑→`FhirClient.CreateAsync` POST→`ReadAsync`로 되조회, code/value/device 일치 확인
- `Bp_panel_posts_with_components_and_subject` — 혈압 panel(85354-9) POST 후 되조회: component 3종 값·subject(Patient/) 참조·device 확인
- `PatientRegistry_creates_retrievable_patient` — PID→Patient 보장 후 그 Patient를 서버에서 되조회

> **통합 테스트가 실제 결함을 잡음**: `PatientRegistry`가 "검색 후 생성"이라, HAPI의 검색 인덱싱 지연 사이 중복 생성을 시도해 `HAPI-2840(duplicate)`으로 거부됐다. → **조건부 생성(`ConditionalCreateAsync`, If-None-Exist)**으로 고쳐 서버가 원자적으로 "있으면 그걸, 없으면 생성"하게 했다. 단위 테스트로는 드러나지 않는 종류의 버그.
>
> 또한 HAPI 공개 서버는 내용 동일 리소스를 중복 거부하므로, 테스트는 실행마다 고유 식별자(장치/환자 id)를 써서 멱등 충돌을 피한다.
